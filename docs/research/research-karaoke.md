### Key technical facts

- **Pitch detection is deliberately cheap in all of these.** USDX: AMDF upgraded to Circular AMDF in PR #461 (merged Aug 2019); plain AMDF had ~12% wrong detections vs CAMDF; CAMDF's octave (double-pitch) errors were accepted precisely because "USDX ignores the octave." UltraStar Play evaluates CAMD correlation only at the ~lag values of each semitone in the singable range (precomputed `halftoneDelays`), i.e., it classifies directly into semitones rather than estimating continuous f0 — plus a `HalftoneContinuationBias` that biases toward the previously detected semitone (cheap temporal smoothing/hysteresis). Min window 256 samples, power-of-two windows, capped max window for performance.
- **Performous is the accuracy outlier**: real-time FFT, precise peak frequencies via phase reassignment, harmonics combined into candidate fundamentals, then temporal filtering. Its detector was good enough that it was ported into Vocaluxe and proposed for librosa. If you want one algorithm to *study* (not copy — GPL), it's this one.
- **The universal scoring recipe (USDX, confirmed in source `UNote.pas`):**
  1. Octave invariance: shift detected tone by ±12 until within 6 semitones of target (`while (Tone - Target > 6) Tone -= 12; ...`) — SingStar does the same ("C3 or C4 both count").
  2. Semitone tolerance by difficulty: hit if `|target − detected| ≤ Range`, `Range = 2 − difficulty` → Easy ±2, Medium ±1, Hard ±0 semitones. UltraStar Play refines Hard to ±0.5 using the continuous frequency, and *rounds/snaps* the sung note to the target when within tolerance (both for scoring and for what's drawn — you see yourself "on" the note).
  3. Per-beat scoring, no explicit timing windows: the UltraStar beat grid is quarter-beats (file BPM = 4× musical BPM), so at 120 real BPM each scoring cell is ~125 ms. Each correct beat of a note earns points; timing tolerance emerges from this quantization. There is no onset-timing judgment at all — sing the right pitch during the note's beats and you score.
  4. Normalized total of 10,000 points: 9,000 distributed across note-beats proportional to note length (golden notes ×2 weight), + up to 1,000 "perfect line" bonus. Rap notes score on any voiced sound; Freestyle notes are unscored.
  5. Forgiveness hacks that make it *feel* fair: UltraStar Play's **joker rule** (a correct beat banks a joker; the next wrong beat is silently converted to correct), and out-of-range/failed pitch detections during a note are *assumed correct* rather than penalized. Performous gives full points anywhere within the correct semitone ("off by a quarter tone and still get the points") with partial credit falloff beyond.
- **Player feel:** typical Performous scores 2,000–9,000 of 10,000 — the ceiling is intentionally unreachable, so improvement is always visible. SingStar's success factors per descriptions: octave invariance, pitch-only (humming scores), instant visual feedback of your pitch bar against the note bar — no lyric/timbre judgment. The genre's lesson: err heavily toward forgiveness; players quit when correct-sounding singing scores 0 (octave errors and mic noise are the top causes).
- **UltraStar .txt format (spec 1.1.0, MIT):** line = `NoteType StartBeat Length Pitch Text`; pitch 0 = C4 (MIDI 60); types `:` normal, `*` golden, `F` freestyle, `R` rap, `G` golden rap, `-` phrase break; header `#BPM` (×4) + `#GAP` (ms offset) define the beat grid. UTF-8, trivially parseable, versioned spec, ecosystem of editors and AI song-creation tools (UltraSinger: Whisper + pitch models + source separation → auto-generates .txt; ultrastar_pitch: NN pitch classification with key-based statistical post-correction — both useful patterns for auto-generating VMatch targets from arbitrary recordings).
- **License wall:** USDX (GPL-2.0), Performous (GPL-2.0+), Vocaluxe (GPL-3.0) — code from these cannot go into a commercial closed product. UltraStar Play + the format spec are MIT — code and spec freely reusable.
- **Smule** (closed): no engineering disclosure found; known publicly via its "Intonation" research dataset (separated vocal tracks, beginner→pro) and the "Deep Autotuner" line of work — i.e., Smule treats pitch deviation-from-reference as its core signal too, and applies data-driven correction. Design lesson: continuous cents-level deviation against a reference performance (not just note grid) enables both scoring and assistive feedback.

### Recommendation

- **Adopt the UltraStar .txt format as VMatch's target-note representation for sung targets. Yes, unambiguously.** MIT-licensed spec, semitone+beat granularity is exactly what a scoring layer needs, huge tooling ecosystem, and AI pipelines (UltraSinger-style: Whisper + source separation + note transcription like Basic Pitch) can auto-generate it from any target recording — which is our "pick any voice clip" flow. Internally, extend it with a parallel continuous f0 curve (cents) precomputed from the target clip, since VMatch's Flow Mode needs contour, not just quantized notes; the .txt is the interchange/Learn-Mode layer, the f0 curve is the visualization layer.
- **Browser MVP scoring:** implement the USDX/Play recipe in TypeScript — it's ~100 lines: fold detected pitch into nearest octave of target (±6 semitones), snap-to-target within a difficulty tolerance (default ±1 semitone, ±2 for casual), score per quarter-beat cell, joker rule, treat detection failures during a note as neutral-positive, normalized 10,000-style total per phrase. Port ideas (or, since it's MIT, actual code) from UltraStar Play's `CamdAudioSamplesAnalyzer`/`PlayerMicPitchTracker`/`PlayerScoreControl` (`UltraStar Play/Packages/playshared/Runtime/PitchDetection/...`). For the pitch detector itself, don't use CAMD — in the browser we can afford better (pYIN/CREPE-class via WASM/onnxruntime-web, a separate research thread); but keep CAMD's two cheap tricks: evaluate/snap on a semitone grid with hysteresis toward the previous note, and octave-fold before comparing.
- **Server-side Learn Mode:** Performous-style analysis is superseded here — use proper offline tools (pYIN/CREPE f0, forced alignment) rather than any karaoke engine. From this corpus, take only the *scoring semantics* (octave folding, tolerance bands, per-beat cells) as the interpretable "pitch/timing" dimensions; nothing in these games handles articulation or timbre at all — that's greenfield for us.
- **Skip:** copying any code from USDX/Performous/Vocaluxe (GPL contamination risk — study algorithms, re-derive from the published papers: AMDF/CAMDF, YIN, FFT reassignment are all public methods); pitch.js (dead since 2013 and GPLv2 — it is an acknowledged port of Performous code, so same copyleft wall); building our own note format.
- **Feel spec to carry forward (the state-of-the-art "fair and fun" recipe):** octave-invariant always-on; full credit within ±~50 cents of the semitone, smooth partial credit to ±1.5–2 semitones (avoid Performous's hard semitone cliff — VMatch's continuous visualization can show sub-semitone convergence instead); no onset-timing punishment, just per-~100ms-cell pitch occupancy; forgiveness mechanics (joker/grace beats, assume-correct on detection dropouts); unreachable score ceiling; and never show a raw 0 for near-misses — show direction (sharp/flat) instead, which is exactly what our living-entity visualization should encode.
