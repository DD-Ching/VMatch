### Key technical facts
- **Proven browser deployment exists (2026)**: parakeet_web runs sherpa-onnx's WASM diarization with the 3D-Speaker CAM++ embedding model (~28 MB) fully client-side in a background worker, WASM int8 by default, WebGPU optional. So chunked speaker embeddings in-browser are demonstrated fact, not speculation.
- **Model size ladder**: CAM++ ~28 MB / WeSpeaker ResNet34-LM 26.5 MB (fp32; int8 = 6.7 MB) → ECAPA ~85 MB → TitaNet ~90 MB → WavLM-base-plus-sv ~380 MB. Only the first tier is acceptable for a web MVP's first-load budget; int8 quantization brings CAM++/ResNet34 to ~7–13 MB.
- **Embeddings are utterance-level by design** (attentive stats pooling over ≥1 s). A useful "streaming" pattern is a 1.5–2 s sliding window re-embedded every 0.5–1 s (~1–2 Hz). CAM++-class models need ~10⁹ FLOPs per 2 s window — tens of ms on WASM-SIMD-threads on a mid-range laptop, i.e. trivially sustainable at 1–2 Hz in a worker. Per-visual-frame (60 fps) neural embedding is **not** realistic or necessary.
- **License minefield is in the weights, not the code**: toolkits are Apache-2.0/MIT across the board, but VoxCeleb-trained checkpoints (WeSpeaker, pyannote's, TitaNet) carry **CC-BY-4.0** — commercially usable with attribution, but VoxCeleb is YouTube-scraped; legal should sign off. WavLM weights are worse: the microsoft/UniSpeech repo license is **CC-BY-SA-3.0** (ShareAlike). 3D-Speaker (Apache-2.0 repo, no separate weights license stated) and ReDimNet/ReDimNet2 (MIT) are the clean-license options. pyannote models add HF gating (breaks anonymous CDN download for a web app). parakeet_web is AGPLv3 — reference only.
- **ONNX export friction**: SpeechBrain and NeMo both require manual, fragile exports (STFT ops). WeSpeaker and 3D-Speaker ship official ONNX. ReDimNet has none yet.
- **transformers.js**: supports HuBERT, UniSpeechSat, PyAnnote (segmentation), and **WavLM** (the D4ve-R/wavlm-base-plus-sv repack is transformers.js-tagged) — but WavLM's ~380 MB class still disqualifies it in-browser. WeSpeaker is **not** in the documented architecture list — so plan on **raw onnxruntime-web** (or sherpa-onnx WASM) rather than transformers.js for the CAM++/WeSpeaker embedding path. The `onnx-community/wespeaker-voxceleb-resnet34-LM` repack (CC-BY-4.0; fp32 26.5 MB + int8 6.7 MB) is worth a 1-day spike.
- **Speaker embeddings measure identity, not "current timbre"**: they're trained to be invariant to content and somewhat to pitch — good for Learn Mode's "how close is your overall voice color", wrong for frame-level Flow Mode dynamics. Cosine similarity needs calibration (same-speaker cosine on these models is typically ~0.4–0.7, not 1.0) — map through a per-model sigmoid/percentile scale before visualizing.
- **Classical per-frame proxies are all trivially computable at 50–100 Hz in JS**: spectral centroid/rolloff/flatness/slope and MFCCs via Meyda (MIT) or ~200 lines of hand-rolled FFT math in the AudioWorklet. HNR: no maintained JS library exists, but an autocorrelation-based HNR (peak ratio at the F0 lag) falls out of the pitch tracker for free — YIN-family aperiodicity is effectively the same signal. Formants: no maintained Praat-quality WASM tracker exists; formantanalyzer.js (Apache-2.0) works but is peak-picking-based and single-maintainer; true LPC (order 12–16 @ 16 kHz, Levinson-Durbin, peak-pick the LPC envelope) is small enough to own ourselves and will behave better on sung vowels.

### Recommendation
- **Flow Mode per-frame timbre proxy (build, don't download)**: compute in our AudioWorklet at the same 50–100 Hz cadence as pitch/energy: log-spectral centroid, spectral slope/tilt, rolloff (85%), spectral flatness (noisiness/breathiness stand-in for HNR), autocorrelation HNR at the tracked F0 lag, and F1/F2 from our own LPC. Use Meyda (MIT) to prototype, then inline the 5–6 extractors we keep — a dependency this small shouldn't gate the render loop. Map (centroid, tilt, flatness/HNR, F1/F2) → visual material properties (brightness, sharpness, grain, hue). Skip formantanalyzer.js as a dependency (single-maintainer, non-LPC method); steal ideas only.
- **Learn Mode utterance-level timbre similarity (browser)**: **3D-Speaker CAM++ int8 ONNX via onnxruntime-web (or sherpa-onnx WASM) in a Web Worker**, lazy-loaded only when the user enters Learn Mode (~7–28 MB). Cleanest license (Apache-2.0), proven in-browser (parakeet_web), half the compute of ECAPA. Validate its English/singing behavior in week 1 — note sherpa-onnx also ships an English VoxCeleb-trained CAM++ export; fallback is WeSpeaker ResNet34-LM (official ONNX, CC-BY-4.0, English VoxCeleb training). Update a smoothed embedding over a 2 s sliding window at ~1 Hz and blend it into the visuals as a slow "voice color convergence" channel — classical proxies carry the fast motion.
- **Server-side / precompute**: target-clip embeddings should be precomputed offline anyway. There, run **WavLM-base-plus-sv or ECAPA** as a second opinion and calibrate the browser model's cosine scale against it — but note WavLM weights are **CC-BY-SA-3.0 (ShareAlike)**, so get legal sign-off or default to ECAPA (Apache-2.0) for this role. Keep an eye on **ReDimNet2 (MIT, ~1M-param B0)** — if we export it to ONNX ourselves it likely becomes the best browser model within this product's lifetime.
- **Skip**: Resemblyzer (unmaintained, 2019-era accuracy), TitaNet (NeMo dependency + shaky ONNX export for no accuracy win we need), pyannote embeddings (gated distribution, just repackaged WeSpeaker), WavLM in-browser (380 MB class), any Praat-WASM plan (doesn't exist; owning LPC is cheaper).
- **Answer to (c)**: yes — chunked/sliding-window speaker embeddings in-browser on a mid-range laptop are realistic in 2026 at ~1–2 Hz with a 7–28 MB int8 model on WASM (SIMD+threads), demonstrably so; continuous 60 fps neural timbre is not, and the two-rate design (fast classical proxies + slow embedding) is the right architecture regardless.
